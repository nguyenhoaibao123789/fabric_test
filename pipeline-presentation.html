04
Silver transformation
Partition by partition.
Resumable by design.
The pipeline never processes everything — only bronze partitions newer than the last watermark. A Python-only notebook reads the native-format bronze file and merges into the Warehouse silver table with SCD Type 2 semantics. The pipeline only orchestrates.
💧
Partition Watermark
ingest_partition in elt_table_config records the last partition successfully loaded to silver. On each run, the pipeline lists bronze partitions, filters to those newer than this value, and processes them in ascending order.
2025-01-10-08 done ✓
2025-01-10-09 done ✓
2025-01-10-10 ← processing now
2025-01-10-11 pending
2025-01-10-12 pending
🔄
Sort via OPENJSON in Warehouse
GetMetadata returns a JSON array of folder names. A Lookup activity passes it through a Warehouse T-SQL query with ORDER BY [name] ASC using OPENJSON(). This guarantees partitions always process oldest-first so the watermark only moves forward.
📓
Python-Only Notebook · SCD2
No PySpark, no %%sql. Reads bronze in its native format with pandas/pyarrow, stages a parquet to Lakehouse, then runs COPY INTO + close-and-insert merge against a Fabric Warehouse silver table via pyodbc.
Python Cell A — Read bronze + add SCD2 row_hash
# Read bronze file in its native format (no conversion in src2brz) if file_type == "parquet": df = pads.dataset(bronze_local, format="parquet").to_table().to_pandas() elif file_type == "csv": df = pd.read_csv(files) elif file_type == "json": df = pd.read_json(files, lines=True) elif file_type == "excel": df = pd.read_excel(files) # SCD2 change detection = SHA-256 over business columns only df["row_hash"] = df[business_cols].apply(_row_to_hash, axis=1) df["valid_from"] = now_utc df["valid_to"] = None df["is_current"] = True
Python → T-SQL Cell B — COPY INTO + SCD2 merge (pyodbc)
# Stage parquet to Lakehouse Files/_staging/, then let Warehouse pull it pq.write_table(pa.Table.from_pandas(df), staging_local) execute(conn, f""" COPY INTO [silver].[{tablename}__stg_{run_id}] FROM '{staging_abfss}' WITH (FILE_TYPE = 'PARQUET') """) # SCD2 close: mark old current rows stale when row_hash changed execute(conn, f""" UPDATE t SET t.valid_to = s.valid_from, t.is_current = 0 FROM [silver].[{tablename}] t JOIN [silver].[{tablename}__stg_{run_id}] s ON {pk_join} WHERE t.is_current = 1 AND t.row_hash <> s.row_hash """) # SCD2 insert: new versions (changed PKs + brand-new PKs) execute(conn, f""" INSERT INTO [silver].[{tablename}] ({col_list}, valid_from, valid_to, is_current) SELECT s.*, s.valid_from, NULL, 1 FROM [silver].[{tablename}__stg_{run_id}] s LEFT JOIN [silver].[{tablename}] t ON {pk_join} AND t.is_current = 1 WHERE t.[{pk}] IS NULL """)

Metadata model
One table.
Every pipeline decision.
All orchestration behaviour is read from elt_table_config at runtime. No pipeline JSON changes needed to add, remove, or modify a table.
elt_table_config primary config — one row per table per layer

Which medallion layer processes this row. Coordinator reads this to route to the right get_config pipeline.

Drives the Switch activity in process_object. Determines which function pipeline fires for this table.
parquet_none_header, csv, parquet, json, excel

Processing order within a run. Tables with the same sequence can run in parallel (src2brz) or sequentially (bronze2silver). Lower numbers run first.

The last partition successfully processed. On the next run, only partitions greater than this value are processed. Updated by update_load_status SP after each successful partition.

06
Execution model
Sequences enforce
dependency ordering.
sequence_number groups tables into batches. Batches run one after another — a higher sequence never starts until all tables in the lower sequence finish.
get_config → ForEach sequence (sequential outer loop)
Sequence 1 — process_object call #1 runs first
orders customers products suppliers
↓ waits for all Sequence 1 tables to complete
Sequence 2 — process_object call #2 depends on seq 1
order_lines transactions
↓ waits for all Sequence 2 tables to complete
Sequence 3 — process_object call #3 depends on seq 2
daily_summary
How it works in the pipeline: get_config does a distinct sequence_number query, then calls process_object for each — passing the full table_config array plus the current sequence_number. process_object filters to matching rows internally.
Within-sequence parallelism varies by layer
Tables within the same sequence don't necessarily run sequentially — it depends on which layer's process_object runs them.
LayerInner ForEachEffect
src2brzparallelAll tables in a sequence copy simultaneously
bronze2silversequentialTables process one at a time within a sequence
How to use sequence_number
Set sequence_number = 1 for base tables with no upstream dependencies. Set sequence_number = 2 for tables that JOIN or depend on sequence 1 outputs. Tables with no cross-dependencies can all share the same sequence number and run in the same batch.
Tip: how process_object filters
The entire table_config array is passed to each process_object call. It uses a Filter activity (@equals(item().sequence_number, pipeline().parameters.sequence_number)) to select only its batch. This avoids a separate config query per sequence.

08
Design principles
Built-in guarantees.
Pipelines orchestrate only.
No transformation logic lives inside pipeline activities — every piece of data work is in a notebook or stored procedure.
🎛️
Metadata-Driven
Adding a new table requires no pipeline changes — just a new row in elt_table_config with the right bronze_file_format, layer, and source details. The coordinator routes it automatically.
Zero pipeline edits
🧮
Orchestrate, Don't Transform
Fabric Pipeline activities (Copy, ForEach, Switch) only move control flow. All data movement and transformation is delegated to notebooks (when data crosses the Lakehouse/Warehouse boundary) or stored procedures (when it's warehouse-to-warehouse). Pipeline JSON stays small; logic stays in version-controlled code.
Notebooks + SPs do the work
💾
Resumable Watermark
ingest_partition is updated after each successful partition — not after a full run. A mid-run failure resumes from the last successful partition on the next trigger, with no data re-processed.
Per-partition commits
📁
Format Preservation
File sources land in bronze exactly as they arrived (csv, json, parquet, xml, excel). The file_type is passed to the Python notebook so the right pandas reader is picked — parsing only happens at the bronze→silver stage.
Bronze = raw truth
🔌
Gateway Support Kept
The copy_from_sql_to_parquet pipeline uses a Copy activity — the only Fabric component that supports on-premise data gateways. Notebooks cannot reach on-prem directly.
On-prem compatible
🔢
Sequence-Controlled Ordering
The sequence_number column groups tables into batches. Batches run in strict order — higher sequences never start until all lower-sequence tables complete, enabling safe cross-table dependency management.
Dependency-aware
🪄
Right Tool for the Layer
Notebooks only appear where data must cross a format boundary — bronze (Lakehouse files) → silver (Warehouse). Once data is inside the Warehouse, silver→gold and gold→platinum use plain stored procedures: no pandas roundtrip, no pyodbc, no Entra token plumbing — just T-SQL the Warehouse runs natively.
SP for warehouse-internal
